# 第一部分：核心改革方向

## 📋 核心改革方向

### 1. 對話記憶系統升級

- **記錄容量**：無限制（使用數據庫存儲）
- **存儲位置**：PostgreSQL/Firestore（按 session_id 分組）
- **上下文範圍**：最近 10 輪對話 **僅限當前 session**（防止跨遊戲污染）
- **隔離機制**：每個 session 獨立，session 結束後存檔

### 2. Token 優化

- **當前估算**：每輪對話 ~500-800 tokens
  - System Instruction: ~200 tokens
  - Context (10 輪): ~300 tokens
  - User Input: ~50 tokens
  - 總計: ~550 tokens/輪
- **優化目標**：降至 ~400 tokens/輪（壓縮 context）

### 3. RAG 集成

- **當前狀態**：未集成
- **改革方案**：
  - 集成真實騙案數據庫（Chroma/Pinecone）
  - 每次 LLM 調用前檢索相關案例
  - 注入到 System Instruction 中
  - 提升騙術/防騙的現實性和精準度

### 4. 回應長度控制

- **改革方案**：在 LLM prompt 中明確要求
  - 「請在 80 字以內回應，不要超過」
  - 使用 token 計數器驗證
  - 後處理時智能截斷（保留完整句子）

### 5. 專家口語化

- **改革方案**：
  - 移除格式化標記（「**分析：**」等）
  - 使用自然對話風格
  - 例如：「呢個係詐騙嚟，你聽我講...」而非「【分析】這是詐騙」

### 6. 騙術/防騙提取升級

- **當前方式**：關鍵字匹配（死板）
- **改革方案**：
  - 調用 LLM 分析回應
  - LLM 判斷使用了哪些騙術/防騙方向
  - LLM 給出評分（1-20）
  - 然後才更新評分系統

### 7. 評分系統重構

#### A. 騙徒計分（基於受害者反應）

- **邏輯**：不看騙徒說什麼，看受害者反應
- **流程**：
  1. 騙徒說話
  2. 受害者反應
  3. LLM 分析：「受害者這個反應，騙徒用了什麼騙術最有效？」
  4. LLM 返回：騙術方向 + 評分 (1-20)
  5. 更新騙徒信用度

#### B. 專家計分（基於受害者反應）

- **邏輯**：不看專家說什麼，看受害者反應
- **流程**：
  1. 專家說話
  2. 受害者反應
  3. LLM 分析：「受害者這個反應，專家用了什麼防騙方向最有效？」
  4. LLM 返回：防騙方向 + 評分 (1-20)
  5. 更新專家信用度

#### C. 受害者反應計分（加分制）

- **規則**：
  - 完全相信 → +騙徒信用度 (1-20)
  - 有點相信 → +騙徒信用度 (1-10)
  - 懷疑 → +專家信用度 (1-20)
  - 拒絕 → +專家信用度 (1-20)
  - **可同時加兩者**（例如：「我有點相信，但也有點懷疑」）
  - **上限**：每次最多 +20

#### D. 警覺性計算

- **公式**：`警覺性 = 專家信用度 - 騙徒信用度`
- **範圍**：-100 ~ +100（標準化為 0-100）
- **含義**：
  - 0-30：低警覺（容易被騙）
  - 30-70：中等警覺
  - 70-100：高警覺（難以被騙）

### 8. 評估系統升級

#### A. 受害者評估

- **何時記錄**：玩家扮演受害者時
- **內容**：
  - 受害者的警覺性
  - 受害者對騙徒/專家的信任度
  - 受害者的反應類型
  - 受害者的脆弱點

#### B. 騙徒評估

- **何時記錄**：玩家扮演騙徒時
- **內容**：
  - 騙徒使用的騙術方向
  - 騙術的有效性
  - 騙徒的策略評分
  - 騙徒的改進建議

#### C. 專家評估

- **何時記錄**：玩家扮演專家時
- **內容**：
  - 專家使用的防騙方向
  - 防騙的有效性
  - 專家的策略評分
  - 專家的改進建議

#### D. 自動訓練模式

- **何時啟用**：系統自動生成對話
- **記錄方式**：同時記錄三個評估
  - 受害者評估
  - 騙徒評估
  - 專家評估

### 9. 關鍵字勝負判定升級

- **當前方式**：硬編碼關鍵字列表
- **改革方案**：
  - LLM 分析是否真的達到勝利條件
  - 例如：「銀行密碼」不一定勝利
    - ✅ 「請提供你的銀行密碼」→ 騙徒勝利
    - ❌ 「你真係想要密碼」→ 不是勝利
    - ✅ 「銀行密碼 123456」→ 騙徒勝利
  - 同理「報警」不一定勝利
    - ✅ 「立即撥打 18222 報警」→ 專家勝利
    - ❌ 「我有朋友報案過」→ 不是勝利
  - LLM 判斷邏輯：
    1. 檢查是否包含關鍵字
    2. 分析上下文語境
    3. 判斷是否真的達到勝利條件
    4. 返回：勝利/未勝利 + 理由

**詳細例子請參考第二部分**

## 🎯 關鍵字勝負判定升級 - 詳細例子

## 核心原則
- **舊方式**：硬編碼關鍵字 → 直接勝利/失敗
- **新方式**：LLM 分析語境 → 判斷是否真的達到勝利條件

---

## 📌 原有 3 組例子

### 例子 1：銀行密碼（騙徒勝利）

```
騙徒：「請提供你的銀行密碼」
受害者：「好的，我的密碼是 123456」

❌ 舊系統：檢測到「密碼」→ 騙徒勝利
✅ 新系統：
  - LLM 分析：受害者真的提供了密碼
  - 判定：騙徒勝利 ✓
  - 理由：「受害者主動提供敏感資料，騙徒達到目標」
```

### 例子 2：銀行密碼（非勝利）

```
騙徒：「你真係想要密碼」
受害者：「我唔會俾你密碼」

❌ 舊系統：檢測到「密碼」→ 騙徒勝利 ❌ 錯誤
✅ 新系統：
  - LLM 分析：受害者明確拒絕
  - 判定：未勝利
  - 理由：「雖然提到密碼，但受害者拒絕提供，騙徒未達目標」
```

### 例子 3：報警（專家勝利）

```
專家：「立即撥打 18222 報警」
受害者：「好的，我現在就打」

❌ 舊系統：檢測到「18222」→ 專家勝利
✅ 新系統：
  - LLM 分析：受害者同意報警
  - 判定：專家勝利 ✓
  - 理由：「受害者同意採取防騙行動，專家達到目標」
```

## 📊 LLM 判定邏輯

```python
class VerdictAnalyzer:
    """LLM 勝負判定分析器"""
    
    async def analyze_verdict(self, message: str, role: str) -> Dict:
        """分析是否達到勝利條件"""
        
        prompt = f"""
分析以下消息是否達到勝利條件：

消息：「{message}」
角色：{role}

勝利條件：
- 騙徒勝利：受害者提供敏感資料（密碼、帳號、驗證碼）或轉賬
- 專家勝利：受害者同意報警或停止與騙徒對話

密碼判定標準：
- ✅ 純數字（4 位以上）
- ✅ 純英文（4 個字母以上）
- ✅ 純符號（3 個符號以上）
- ✅ 混合（任意組合）

帳號判定標準：
- ✅ 純數字（10-12 位）
- ✅ 英文+數字
- ✅ 數字+符號
- ✅ 任意組合

轉賬判定標準：
- ✅ 已經轉賬（完成時態）
- ✅ 正在轉賬（進行時態）
- ✅ 轉賬完成（確認時態）
- ✅ 已經扣款

報警判定標準：
- ✅ 已經報警（完成時態）
- ✅ 正在報警（進行時態）
- ✅ 已經撲打 18222
- ✅ 已經聯絡警察

請分析：
1. 是否包含關鍵字？
2. 是否是真實行動（不是假設、不是別人的經歷）？
3. 是否達到勝利條件？

返回 JSON：
{{
    "verdict": "scammer_win" | "expert_win" | "ongoing",
    "confidence": 0.95,
    "reasoning": "受害者提供了看起來真實的銀行密碼",
    "keyword_detected": "銀行密碼",
    "context_analysis": "受害者主動提供，完成時態",
    "is_real_action": true
}}
"""
        
        result = await llm.generate(prompt)
        return json.loads(result)
```
