## 📊 性能影響

### Token 使用

- **修復前**: ~50-100 tokens/請求
- **修復後**: ~500-1000 tokens/請求 (包含10輪歷史)

### 響應時間

- **預期增加**: 0.5-1.5 秒 (取決於歷史長度)
- **原因**: 需要處理更多 tokens
- **優化**: 只取最近 10 輪 (可調整)

### 控制歷史長度

如需調整歷史長度，修改這行：

```python
for msg in conversation_history[-10:]:  # ← 改為 -5, -15 等
```

## 📝 技術細節

### /api/generate 的上下文處理

Ollama 的 `/api/generate` 端點：
- ✅ **支持上下文**: 通過 `prompt` 字段包含歷史
- ✅ **格式**: 
  ```
  用戶: 消息1
  助手: 回應1
  用戶: 消息2
  ```
- ✅ **我們的實現**: 自動構建這種格式

### 為什麼取最近 10 輪？

1. **Token 限制**: 避免超過模型上下文窗口
2. **相關性**: 太久遠的對話不重要
3. **性能**: 減少處理時間
4. **平衡**: 10輪足夠保持連貫性
